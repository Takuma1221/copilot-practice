# Copilot SDK 学習計画

## 目的

この計画は、Copilot CLI の初歩を終えた段階から、Copilot SDK を最小構成で理解し、このリポジトリに合う小さな read-only ツールを作れるようになることを目的とする。

今回のゴールは3つに絞る。

- Copilot SDK の役割を、自分の言葉で説明できる
- SDK の最小サンプルを動かせる
- このリポジトリ向けの小さなツール案を1つ選び、実装準備まで進められる

## 今の前提

現状の理解として、以下はすでに満たしている。

- Copilot CLI を起動できる
- `copilot --version` が通る
- interactive mode と `copilot -p` の違いを理解している
- skills / agents / MCP の概念はすでに理解している

つまり、SDK 学習に入る前提は十分に整っている。

## SDK を今学ぶ理由

CLI は「Copilot を使う側」の体験が中心だったが、SDK は「Copilot を組み込む側」に回れる。
今感じている「チャットとの差が薄い」「エンジニアっぽさが弱い」という感覚は、SDK に入ることで解消しやすい。

SDK に進む価値は次の3点にある。

- 入出力を自分で設計できる
- 結果を JSON など機械可読にできる
- 将来的に CLI、スクリプト、エージェント設計と接続できる

## 学習方針

最初から大きなものは作らない。順番は固定する。

1. SDK の最小構成を理解する
2. 最初のメッセージ送信を動かす
3. streaming を確認する
4. このリポジトリ向けの read-only ツールを1つ設計する
5. 実装前に入出力を決める

最初は「賢いアプリを作る」ではなく、「最小の動くものを正しく理解する」が優先。

## 学習フェーズ

### Phase 1. SDK の全体像を掴む

期間の目安: 半日

この段階で理解すべきこと:

- Copilot SDK は Copilot を使ったアプリケーションを作るための土台である
- CLI の認証が前提として使われる
- Node.js 18 以上が前提になる
- 最初は TypeScript で進めるのが無難

この段階の到達基準:

- CLI と SDK の違いを説明できる
- SDK は「会話する道具」ではなく「Copilot を組み込む実装手段」だと理解している

### Phase 2. 最小サンプルを動かす

期間の目安: 1日

やること:

- 検証用ディレクトリを切る
- npm 初期化をする
- `@github/copilot-sdk` と実行補助を導入する
- 最初のメッセージ送信を行う
- 応答を標準出力に出す

この段階では、まだこのリポジトリ固有の機能を作らない。
目的は「SDK が本当に動く」「CLI 認証がつながっている」と確認することだけ。

この段階の到達基準:

- サンプルが一度でも動く
- コードのどこが入力で、どこが出力か説明できる

### Phase 3. streaming と出力制御を理解する

期間の目安: 半日

やること:

- 通常応答と streaming の違いを確認する
- 単なる文章ではなく、構造化された出力を求める
- JSON で返す前提の設計を試す

この段階で見るべきこと:

- なぜ将来のツール化に JSON が向くのか
- どの粒度で出力スキーマを切ると扱いやすいか

この段階の到達基準:

- 「人間向けの返答」と「機械向けの返答」を分けて考えられる

### Phase 4. このリポジトリ向けのツール案を決める

期間の目安: 半日

最初の題材候補は次の3つに限定する。

1. Markdown 記事レビュー補助ツール
2. TEMPLATE.md 準拠チェックツール
3. research/ 配下の参考資料候補抽出ツール

最初の題材として最も無難なのは 1 か 2。
理由は、read-only で済み、入力と出力が決めやすく、今のリポジトリに密接に関係しているから。

おすすめの第一候補:

- Markdown 記事レビュー補助ツール

理由:

- 既存の article-review 的な観点とつながる
- `articles/final/` を読むだけで成立する
- JSON で findings を返す形に落としやすい

この段階の到達基準:

- 何を作るかを1つに決められる
- なぜその題材を選ぶか説明できる

### Phase 5. 実装前の入出力設計をする

期間の目安: 1日

ここではまだ実装しない。設計だけする。

決めるべきこと:

- 入力は何か
- 出力は何か
- 何をしないか
- どこまでを SDK 側に持たせるか

例として、Markdown 記事レビュー補助ツールなら次のように置ける。

- 入力: 対象ディレクトリ、レビュー観点、テンプレートファイル
- 出力: `file`, `severity`, `issue`, `reason`, `suggestion` を持つ findings 配列
- 非対象: 自動修正、ファイル上書き、公開処理

この段階の到達基準:

- 実装前に API 的な形を言葉にできる
- 副作用を持たせない理由を説明できる

## 最初の検証ディレクトリでやること

まだ作成前だが、検証ディレクトリは役割を限定する。

目的:

- SDK の最小サンプルを隔離して試す
- 既存のリポジトリ本体にすぐ混ぜない
- 失敗しても捨てやすくする

このディレクトリでは最初に次だけやる。

- 最小サンプルを動かす
- streaming を試す
- JSON 出力の雛形を試す

まだやらないこと:

- 本番用ディレクトリ構成に寄せる
- 複数ツール化する
- MCP サーバー化する
- ファイル自動修正を入れる

## 1週間の進め方

### Day 1

- SDK の役割を整理する
- 公式 quickstart 前提の最小構成を把握する
- 何を作るか候補を3つ見る

### Day 2

- 最小サンプルを動かす
- CLI 認証との接続を確認する

### Day 3

- streaming を確認する
- JSON 出力を試す

### Day 4

- このリポジトリ向けの第一題材を決める
- 入出力を紙に書けるレベルまで整理する

### Day 5

- 実装対象を固定する
- 検証ディレクトリを作る準備をする

## やらない方がよいこと

今の段階で避けるべきことは次の通り。

- いきなり大きなアプリを作る
- いきなりファイル自動修正を入れる
- いきなり SDK と CLI と MCP を全部混ぜる
- 最初から汎用ツールにしようとする
- 出力形式を曖昧にしたまま実装を始める

## 学習完了の目安

この計画の完了時に、以下が言えれば十分である。

- SDK は何のために使うか
- このリポジトリで最初に作るべき題材は何か
- そのツールの入力と出力は何か
- なぜ最初は read-only にするべきか
