# GitHub Copilot CLI / SDK 学習計画書

## 目的

このプロジェクトで進めている GitHub Copilot 学習を、カスタマイズ理解から実運用理解へ進める。
特に以下を到達目標とする。

- Copilot CLI を安全に使い、ターミナル作業の補助に組み込める
- Copilot SDK の最小実装を理解し、小さなツールを自作できる
- 既存の skills / agents / MCP 理解と CLI / SDK を切り分けて説明できる

## 現状プロジェクトから見た理解度の推定

### できていること

このリポジトリには、Copilot のカスタマイズ概念をかなり整理して扱えている痕跡がある。

- AGENTS.md で orchestrator / writer / publisher の役割分担が整理されている
- skills に手順、受け入れ基準、レビュー観点が定義されている
- architecture.md でエージェント実行フローが可視化されている
- article.writer で Tavily MCP、article.publisher で Notion MCP という外部連携の発想が使えている
- articles/final/ 配下の成果物を見る限り、テンプレート駆動で記事作成ワークフローを回す発想が定着している

### まだ弱いこと

一方で、CLI と SDK の実装経験はまだ浅いと判断できる。

- Copilot CLI の実行ログ、運用メモ、手元のコマンドパターンはまだリポジトリ上に見当たらない
- Copilot CLI の programmatic mode、tool approval、trusted directories、hooks などの運用要素は未導入
- Copilot SDK の依存関係やサンプルコードがまだ存在しない
- 自作スクリプトを Copilot から呼びやすい形に整える練習はこれから
- MCP は使う側の理解が中心で、作る側の理解にはまだ踏み込んでいない

### 推定レベル

- Copilot カスタマイズ概念理解: 中級
- Copilot CLI 実務利用: 初級手前
- Copilot SDK 実装: 未経験に近い初級

要するに、概念は先行して理解しているが、CLI と SDK の実行体験が追いついていない状態である。

## 学習方針

学ぶ順序は Copilot CLI を先、Copilot SDK を後にする。

理由は3つある。

- CLI の方が短いフィードバックループで「Copilot に実行させる感覚」を掴みやすい
- SDK は CLI の認証や実行文脈を前提にする場面がある
- このリポジトリでは、まず既存ワークフローを CLI で補助し、その後 SDK で小さく拡張する方が自然である

## 学習の全体像

### Phase 0. 用語の切り分け

最初に以下を1回で言い分けられるようにする。

- Chat: 相談、説明、コード提案が中心
- CLI: ターミナルから Copilot を使う実行補助
- SDK: Copilot を使ったアプリやツールを自作するための開発キット
- Skills / Agents: Copilot の振る舞いを誘導・構成するカスタマイズ
- MCP: Copilot に外部ツールやデータソースを渡す接続方式

この段階では実装しなくてよい。混同しないことが目標。

## 学習計画

### Step 1. Copilot CLI の基礎を体験する

期間の目安: 2日

学ぶこと:

- インストール方法
- 認証方法
- interactive interface と programmatic interface の違い
- ask/execute と plan mode の違い
- `/model`、`/context` など基本操作
- tool approval の考え方

このプロジェクトでやる演習:

- このリポジトリ直下で Copilot CLI を起動する
- articles/final/ 配下の記事一覧を要約させる
- architecture.md を読ませてワークフローを要約させる
- TEMPLATE.md に対して final 記事のテンプレ欠落を指摘させる

この段階の到達基準:

- CLI がチャットの代替ではなく、ターミナル作業に乗る道具だと説明できる
- CLI に安全確認付きでローカル読取系タスクを任せられる

### Step 2. Copilot CLI を日常作業に落とし込む

期間の目安: 3日

学ぶこと:

- trusted directories の意味
- `--allow-tool` / `--deny-tool` / `--allow-all-tools` の違い
- programmatic mode での単発利用
- custom instructions の追加
- 必要なら hooks や plugins の存在を把握する

このプロジェクトでやる演習:

- final 記事3本を対象に「根拠URL不足」「更新日欠落」「テンプレ逸脱」を検査するプロンプトを作る
- grep や git を使うタスクを CLI に依頼し、承認付きで実行させる
- CLI 用の再利用プロンプトを3本作る
  - 記事レビュー用
  - リポジトリ要約用
  - 最近の変更確認用

成果物:

- 自分用の CLI プロンプト集
- 何を許可し、何を許可しないかの運用ルール

この段階の到達基準:

- CLI を安全に使う条件を説明できる
- このリポジトリで CLI を使うべき場面を3つ挙げられる

### Step 3. Copilot SDK の最小構成を理解する

期間の目安: 2日

前提:

- Node.js 18 以上
- Copilot CLI が利用可能であること
- `copilot --version` が通ること

学ぶこと:

- Copilot SDK は technical preview であること
- `@github/copilot-sdk` を npm で導入する流れ
- 最初のメッセージ送信
- streaming response の基本
- CLI 認証と SDK 利用のつながり

おすすめの進め方:

- 新しい検証用ディレクトリを切る
- TypeScript で最小サンプルを動かす
- まずは「メッセージを送り、応答を受け取る」だけに集中する

この段階の到達基準:

- SDK が「Copilot を使うアプリを作るための土台」だと説明できる
- 最小のサンプルコードを手元で動かせる

### Step 4. SDK で小さなツールを1つ作る

期間の目安: 4日

最初の題材は、このリポジトリに寄せる。

候補:

- Markdown 記事レビュー補助ツール
- TEMPLATE.md 準拠チェックツール
- research/ 配下から参考記事候補を抽出するツール

最初の実装要件:

- 入力が明確であること
- 出力が機械可読であること
- まずは read-only であること
- 副作用を持たせないこと

具体例:

- 入力: 対象ディレクトリ、テンプレートファイル、レビュー観点
- 出力: JSON の findings 配列
- 処理: 各 Markdown を走査して欠落項目や弱い箇所を返す

この段階の到達基準:

- 「単なるスクリプト」と「Copilot から使いやすいツール」の違いを説明できる
- 自作ツールの入出力を定義できる

### Step 5. 現在のエージェント設計と SDK / CLI の役割分担を整理する

期間の目安: 2日

この段階で、今のプロジェクトを次のように分解して考える。

- skills / agents: 振る舞いとワークフローの設計
- CLI: その場での調査、レビュー、Git 操作、スクリプト実行
- SDK: 再利用可能な補助機能の実装
- MCP: 外部サービスや外部ツールへの接続

このプロジェクトで整理すべき問い:

- 記事レビューは CLI で十分か、それとも SDK ツール化すべきか
- Notion 連携は MCP 利用で十分か、それとも別実装が必要か
- article.writer に渡す下準備を CLI にやらせる場面はあるか

この段階の到達基準:

- 「何を CLI でやるか」「何を SDK にするか」を、このリポジトリ文脈で説明できる

## 4週間の実行プラン

### 1週目: CLI に慣れる

- Copilot CLI をインストールして認証する
- 読み取り専用タスクを中心に10回以上使う
- 承認付き実行の挙動に慣れる
- このリポジトリ向けの定番プロンプトを3本作る

### 2週目: CLI を運用レベルにする

- tool approval の使い分けを試す
- 記事レビュー系の作業を CLI に寄せる
- trusted directories と安全運用ルールを自分の言葉でまとめる

### 3週目: SDK の最小実装を動かす

- 検証用ディレクトリで SDK quickstart を実施する
- 最初のメッセージ送信と streaming を動作確認する
- SDK の実行構造をメモ化する

### 4週目: このプロジェクト向けに小さく実装する

- Markdown 記事レビュー補助ツールを作る
- 出力を JSON にする
- article ワークフローのどこに差し込めるか整理する

## この順番で学ぶ理由

今のあなたは、skills / agents / MCP の概念は理解している。一方で、CLI と SDK の理解が弱いのは「実行感覚」と「実装感覚」が不足しているからである。

そのため、学習順は以下が最も自然である。

1. CLI で Copilot にローカル作業をさせる
2. その制御と安全性を理解する
3. SDK で最小のアプリを作る
4. 最後に既存ワークフローへ統合する

逆に、最初から SDK で大きいものを作ると、CLI の前提知識と運用感覚が不足したまま設計だけ先に進みやすい。

## 最初の実践課題

最初にやるべき課題は次の順番がよい。

1. Copilot CLI で articles/final/ の3記事をレビューさせる
2. 同じレビュー観点を自分の言葉でテンプレ化する
3. Node.js または Python で Markdown 一括レビューの read-only スクリプトを書く
4. そのスクリプトを将来 SDK ツール化する前提で、入力と出力を整理する

## やらない方がよい順番

以下は今の段階では後回しでよい。

- いきなり Marketplace 公開を前提にした Copilot Extension を作る
- いきなり remote MCP server を自作する
- CLI の安全設計を飛ばして `--allow-all-tools` 前提で使い始める
- いきなり大規模な自動修正ツールを作る
