# Copilot CLI 学習ガイド

## 目的

このガイドは、[research/2026-03-28-copilot-cli-sdk-study-plan.md](research/2026-03-28-copilot-cli-sdk-study-plan.md) の CLI パートを、今のリポジトリですぐ実践できる形に落としたものである。

最初の1週間で目指すのは次の3点。

- Copilot CLI を安全に起動し、基本操作に慣れる
- このリポジトリの読取系タスクを CLI に任せられるようになる
- チャットと CLI の使い分けを実感で理解する

## 事前確認

最初に確認すること。

1. Copilot CLI がインストールされている
2. `copilot --version` が通る
3. `copilot` を起動してログインできる
4. このリポジトリを trusted directory として扱う判断ができる

## まず覚えること

CLI 初学者は、最初から何でもやろうとしない方がよい。最初は次だけで十分。

- `copilot`
- `/model`
- `/context`
- plan mode と ask/execute mode の違い
- approval prompt の意味

最初の理解は以下で足りる。

- ask/execute: 作業を進めるモード
- plan mode: 実装前に方針を整理するモード
- approval: 実行前に人間が止めるための安全弁

## Day 1: 読み取り専用で慣れる

この日は、書き換えや実行ではなく、読んで要約させることだけやる。

### 演習1

プロンプト例:

```text
このリポジトリの目的を説明して。AGENTS.md と architecture.md を読んで、エージェントの役割分担と処理フローを日本語で要約して。
```

見るポイント:

- AGENTS と architecture の対応関係を拾えているか
- orchestrator / writer / publisher の違いを説明できているか

### 演習2

プロンプト例:

```text
articles/final/ 配下の記事を見て、それぞれのテーマの違いを3行ずつでまとめて。
```

見るポイント:

- 記事ごとの差分を読めているか
- 単なるタイトル読みではなく本文要点を拾えているか

### 演習3

プロンプト例:

```text
TEMPLATE.md と articles/final/ を比較して、テンプレート構成に対して不足しそうな箇所があれば列挙して。
ファイルは変更しないで。
```

見るポイント:

- 変更しない条件を守れるか
- 不足点が具体的か

articles/final/ の記事3本をレビューして。観点は、根拠URL、情報更新日、テンプレ準拠、実用性。致命的な問題だけ先に出して。

## Day 4: programmatic mode を実務レベルで使う

ここからは、単発要約ではなく、CLI の価値が出やすい使い方に寄せる。
狙いは「1回で終わるけれど、チャットより CLI の方が速い仕事」を増やすこと。

### 演習8

コマンド例:

```bash
copilot -p "articles/final/ の3記事を比較して、テーマの重複、読者層の重複、説明の抜けを列挙して。結果は markdown の箇条書きで返して。ファイルは変更しないで。" 
```

目的:

- programmatic mode を分析タスクに使う
- 会話を開かずに、単発のレビュー結果だけを回収する感覚を掴む

### 演習9

コマンド例:

```bash
copilot -p "AGENTS.md、architecture.md、.github/skills/article-main/SKILL.md を読んで、このプロジェクトのワークフロー上のボトルネック候補を3つ挙げて。ファイルは変更しないで。" 
```

目的:

- 複数ファイルを跨ぐ読解を 1 回の実行で済ませる
- 要約ではなく、診断タスクに CLI を使う

## Day 5: 安全なツール許可付き実行を覚える

ここで初めて、read-only のシェルコマンドを条件付きで許可する。
重要なのは、無制限許可ではなく、必要最小限だけ通すこと。

### 演習10

コマンド例:

```bash
copilot -p "research/ 配下で Copilot CLI に関する記述を探して、重要な学習ポイントを3つに絞ってまとめて。必要なら rg を使ってよい。ファイルは変更しないで。" --allow-tool='shell(rg)'
```

目的:

- `--allow-tool` の感覚を掴む
- 読取専用の検索を、承認込みでなく事前許可で回す

### 演習11

コマンド例:

```bash
copilot -p "このリポジトリの最近の変更傾向を知りたい。git log を使って直近のコミットを確認し、学習テーマに関係ある変更だけ要約して。ファイルは変更しないで。" --allow-tool='shell(git log)'
```

目的:

- Git 系の read-only タスクを CLI に任せる
- 許可の粒度を細かく考える習慣を作る

### 覚えるべきこと

- `--allow-all-tools` はまだ使わない
- `shell(rg)` のように絞って許可する
- Git も `git` 全体ではなく `git log` や `git status` のように狭く考える

## Day 6: 出力形式を制御して再利用性を上げる

ここから先は、単なる会話ではなく「後で使える出力」を取る練習をする。
CLI の価値は、結果をそのまま次の処理につなぎやすいことにある。

### 演習12

コマンド例:

```bash
copilot -p "articles/final/ の記事3本をレビューして。出力は JSON で返して。キーは file, severity, issue, reason, suggestion。ファイルは変更しないで。" 
```

目的:

- 機械可読な出力を要求する練習
- 将来のスクリプト化、SDK 化につながる発想を作る

### 演習13

コマンド例:

```bash
copilot -p "articles/final/nextjs-overview.md を初心者向け記事としてレビューして。改善点を high, medium, low の3段階で分類して返して。ファイルは変更しないで。" 
```

目的:

- 出力形式を固定する練習
- あとで自分用テンプレートに落とし込みやすくする

## Day 7: 高報酬課題をやる

ここは、CLI を使う意味がはっきり出る課題にする。
単なる要約ではなく、「自分の仕事を減らす」タスクに寄せる。

### 課題A: ミニ設計レビュー

コマンド例:

```bash
copilot -p "AGENTS.md、architecture.md、.github/agents/article.orchestrator.agent.md、.github/agents/article.writer.agent.md を読んで、このワークフローの設計上の弱点を3つ挙げて。各指摘には、なぜ弱いかと、最小の改善案を付けて。ファイルは変更しないで。" 
```

狙い:

- 複数ファイルを跨いだ設計評価
- チャットより CLI の方が速い単発レビューの体感

### 課題B: 変更候補の下準備

コマンド例:

```bash
copilot -p "research/2026-03-28-copilot-cli-practice-guide.md を改善したい。上級者向けとして弱い箇所を3つ指摘し、差し替え候補の見出し案だけ返して。まだ編集しないで。" 
```

狙い:

- 自分のドキュメント改善の下調べを CLI に任せる
- いきなり編集させず、先に材料だけ取る運用に慣れる

### 課題C: 自分用プロンプト資産を作る

最低限、次の3本は作る。

#### 1. 設計レビュー用

```text
このリポジトリの設計をレビューして。対象は AGENTS.md、architecture.md、.github/agents 配下。問題点は重要度順に列挙し、各項目に最小の改善案を付けて。ファイルは変更しないで。
```

#### 2. 記事レビュー用

```text
articles/final/ 配下の記事をレビューして。観点は、事実性、根拠URL、更新日、テンプレ準拠、実用性。まず重要度の高い問題から列挙して。必要なら出力を JSON で返して。ファイルは変更しないで。
```

#### 3. 変更確認用

```text
このリポジトリの最近の変更点を確認したい。必要なら git log や git status を使って、学習テーマに関係ある変更だけ要点をまとめて。ファイルは変更しないで。
```

## CLI 学習中のルール

最初の1週間は次を守る。

- いきなり `--allow-all-tools` は使わない
- まずは read-only タスク中心で使う
- 編集や削除より、検索、要約、比較、レビューから始める
- approval prompt で出たコマンドは意味を読んでから許可する
- ホームディレクトリ直下では不用意に起動しない

## つまずきやすい点

### 1. チャットとの差がわかりにくい

最初は差が小さく見えるが正常である。CLI の価値は、ターミナル文脈で連続的に試せることにある。

### 2. 何を任せていいかわからない

最初は「読む」「探す」「まとめる」「レビューする」に限定すると失敗しにくい。

### 3. いきなり危ない操作をしそうになる

削除、上書き、Git push は後回しでよい。最初は権限管理の感覚を作ることが先。

## このガイドのゴール

このガイドが終わるころに、以下が言えれば十分である。

- CLI はチャットの代わりではなく、単発の実務タスクを速く回す道具である
- このリポジトリなら、レビュー、検索、設計評価、Git 確認、出力整形に使える
- `--allow-tool` を絞って使う感覚がある
- 後でスクリプトや SDK に接続できる出力形式を意識できる
