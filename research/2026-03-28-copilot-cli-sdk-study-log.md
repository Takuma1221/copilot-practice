## 今日理解できたこと

### CLI は「ターミナルで Copilot を使う」もの

CLI は、チャットの代替というより、ターミナル作業に乗る道具だった。

今日の時点で理解できたこと:

- `copilot` は対話モード
- `copilot -p "..."` は単発実行モード
- `/model` は使うモデルの切り替え
- `/context` は会話コンテキストの使用状況確認

ただし、CLI 単体では「賢いチャットを端末で使っている」感覚からは抜けにくかった。

### SDK は「Copilot を組み込む」もの

SDK は、Copilot を自分のコードから呼び出し、ワークフローとして固定できるのが大きかった。

特に今日作った記事レビュー実装では、以下をコードとして明示できた。

- どのファイルを読むか
- どんなプロンプトを送るか
- どんな JSON 形式で返してほしいか
- 壊れた JSON をどう救済するか
- 複数ファイル結果をどう集計するか

この「処理の流れをコードで固定できること」が、SDK の強さだと理解した。

## チャット / CLI / SDK の違い

### チャットの強み

- 相談しやすい
- 概念理解や壁打ちに向く
- コード断片や設計方針の説明をその場で受けやすい

チャットが強い理由:

- 人間が曖昧な問いを投げても成立しやすい
- 途中で前提を変えながら会話を続けやすい
- 結論より思考整理に向いている

チャットが向く使い方:

- まず何を作るべきか相談する
- エラーの意味を聞く
- 設計案を比較する
- コードの意味を理解する

### CLI の強み

- ターミナル文脈にそのまま乗る
- 単発タスクを速く回せる
- Git、検索、ログ確認など既存コマンドと相性が良い
- shell script と組み合わせて半自動化しやすい

CLI が強い理由:

- 最終的に欲しいものが「実行可能なコマンド」である場面に強い
- 一時的な確認や調査を会話より短い距離で回せる
- シェルの資産をそのまま活かせる

CLI が向く使い方:

- Git 履歴や差分の確認
- grep や rg を使った検索補助
- ログ確認
- 単発レビューや要約
- shell script と合わせた実務運用

### SDK の強み

- 処理の流れをコードとして残せる
- 入出力を固定できる
- JSON や型で後続処理につなげやすい
- セッションや役割を分離したワークフローを書きやすい
- チャットや shell script より可読性が高い

SDK が強い理由:

- 「毎回人が同じ依頼をする」部分をプログラムへ移せる
- 出力形式を決め打ちし、検証と再試行を書ける
- TypeScript / Python の構造化されたコードとして保守できる

SDK が向く使い方:

- 複数ファイルを同じルールで処理する
- レビュー結果を JSON で返す
- セッション分離を明示したい処理
- ワークフローを再利用したい処理
- 完成品を教材として読み返したい処理

### 今日の理解での差分

チャットは「考えるための場」に強い。

CLI は「その場で回す作業」に強い。

SDK は「処理を仕組みとして残す」ことに強い。

この3つは競合というより、段階ごとに役割が違う。

## shell script と比べたときの SDK の良さ

今日の理解では、SDK の優位性は次のように整理できる。

### 1. 可読性

shell script でも複数処理をつなげられるが、条件分岐やエラー処理、JSON 整形、再試行、集計が増えるほど読みづらくなる。

今回の [sdk-lab/article-review-tool/src/review.ts](sdk-lab/article-review-tool/src/review.ts) は TypeScript なので、役割ごとに関数を切り出しやすく、処理の責務が見やすい。

### 2. 型と構造

SDK 側では [sdk-lab/article-review-tool/src/types.ts](sdk-lab/article-review-tool/src/types.ts) で JSON スキーマを固定できた。

shell script でも JSON は扱えるが、型や構造の検証は弱くなりやすい。SDK だと Zod のような手段で結果を検証できる。

### 3. セッション分離

今回のレビュー実装では、1ファイルごとに別セッションを張っている。
これにより、レビュー文脈が混ざりにくく、ファイル単位の処理として分離できた。

### 4. 失敗時の救済が書きやすい

モデルが壊れた JSON を返したとき、JSON 修復の再問い合わせを 1 回だけかける処理を入れた。
こういう救済は shell script でも可能だが、SDK の方が意図をコードで表現しやすい。

## 今日作った実装から見えた SDK のメリット

今回の [sdk-lab/article-review-tool/src/review.ts](sdk-lab/article-review-tool/src/review.ts) から見えたメリットは次の通り。

- 記事レビューというワークフローを 1 本のコードに固定できる
- プロンプト、入力ファイル、出力形式を明示できる
- セッションをファイル単位で分けてレビュー内容の混線を避けられる
- モデルの出力をローカルで検証、補正、集計できる

今日の実装はまだ逐次実行だが、将来的には並列化もしやすい。
つまり、CLI で手作業していたレビューを、段階的にアプリ化できる土台になっている。

## SDK / CLI の良い活用方法

今日の理解をもとに、活用方法は「何が強いか」から逆算して考えるのがよい。

### チャットの良い活用方法

#### 1. 要件整理に使う

理由:

- チャットは曖昧な状態からでも会話を始めやすい
- 前提を途中で変えても対応しやすい

向いている例:

- 何を学ぶべきか整理する
- 実装前に設計案を比較する
- 用語や概念を理解する

#### 2. コード読解の補助に使う

理由:

- 「この関数は何をしているか」のような局所的質問に強い
- 流れを自然言語で噛み砕いてもらえる

向いている例:

- `review.ts` の理解
- streaming と non-streaming の差分理解

### CLI の良い活用方法

#### 1. 単発の実務タスクに使う

理由:

- CLI はコマンド実行までの距離が短い
- 結果を見てすぐ次の試行に移れる

向いている例:

- `copilot -p` でリポジトリ要約を取る
- Git 履歴や変更点を確認する
- Markdown ファイルを検索する

#### 2. shell script と合わせて半自動化する

理由:

- CLI はシェルの世界にそのまま乗る
- `rg`、`git`、`jq` などの既存コマンドと自然に組み合わせられる

向いている例:

- レビュー対象ファイルの列挙
- 変更ファイル一覧の取得
- 既存の開発フローに AI 補助を差し込む

#### 3. プロンプトの試作に使う

理由:

- いきなり SDK に落とす前に、依頼文の出来を確かめやすい

向いている例:

- 記事レビュー観点の試作
- 設計レビュー用プロンプトの試作

### SDK の良い活用方法

#### 1. 反復処理をワークフロー化する

理由:

- SDK は「読む → 送る → 検証する → 集計する」をコードで固定できる
- 人手の繰り返しを減らせる

向いている例:

- 複数記事の一括レビュー
- 同一テンプレートの準拠チェック
- ファイルごとのセッション分離が必要な処理

#### 2. 出力を構造化して扱う

理由:

- SDK は JSON 出力や型検証と相性が良い
- モデルの返答をそのまま後続処理へつなげやすい

向いている例:

- findings 配列の収集
- スコアリング結果の保存
- 後で UI や別スクリプトに渡す処理

#### 3. 読める完成品として残す

理由:

- shell script より責務分離しやすい
- 関数、型、設定ファイルに分けて保守できる

向いている例:

- 教材としてのミニツール
- チーム内で共有する補助ツール
- 将来並列化や再試行を足したい処理

### 2つを組み合わせるとよい使い方

#### 1. チャットで整理し、CLI で試し、SDK で定着させる

理由:

- チャットは曖昧な問いを整理しやすい
- CLI は単発検証が速い
- SDK は正しかった流れをコード化できる

使い方:

- まずチャットで観点を整理する
- 次に CLI でプロンプトを試す
- 最後に SDK へ落として仕組みにする

#### 2. CLI は探索、SDK は本番化

理由:

- CLI は実験に向く
- SDK は再利用と保守に向く

使い方:

- 単発レビューは CLI
- 毎回やるレビューは SDK

#### 3. チャットは理解、CLI は運用、SDK は実装

理由:

- 3者で強みが違う
- 1つに寄せるより分担した方が無理がない

使い方:

- 理解したいときはチャット
- 現場で素早く回すときは CLI
- 形に残して広げたいときは SDK

## 今の自分にとっての実践的な結論

今日の時点では、CLI だけだと「端末で賢いチャットをしている」感覚が強かった。
一方で SDK は、完成品のコードを見るとかなりエンジニアリング寄りだった。

特に、以下が自分にとって大きかった。

- `CopilotClient` からセッションを張る流れ
- streaming と非 streaming の差
- ツールがなくてもモデル本体で返答できること
- ワークフローを TypeScript のコードとして整えられること

今後は、CLI を深追いするより、SDK 側で小さい完成品を増やしていく方が学習効率が高そうだと感じた。
